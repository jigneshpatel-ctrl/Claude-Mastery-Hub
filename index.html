Tips & Tricks

Every tip has 3 copy-ready examples — paste directly into Claude and try it now.

Fundamentals

01

Role + Context + Goal in every prompt

The foundation of all good prompts. Claude behaves like whoever you tell it to be — give it a role, the context it needs, and exactly what you want.

3 ready-to-use examples
Backend engineer
You are a senior backend engineer at a fintech startup. We use Node.js, PostgreSQL, and Razorpay. I need you to explain idempotency keys — what they are, how to implement them for payment initiation, and give me production-ready Express middleware pseudocode.
Product manager
You are a senior product manager at a B2B SaaS company. Our product is a CRM for SMBs. I need a concise PRD for a bulk CSV import feature — problem statement, 3 success metrics with targets, 4 user stories, scope, and 3 edge cases.
Tech lead review
You are a tech lead reviewing a system design for a junior engineer. Context: VehicleInfo app, Node.js + PostgreSQL, 500 payments/day. I need you to review this payment flow design and flag any scalability issues, missing error handling, and security gaps. Be direct and specific.
02

Chain-of-Thought on complex tasks

Adding Think step by step before answering forces Claude to reason through logic before concluding. Single highest-leverage technique — quality difference is dramatic on complex problems.

3 ready-to-use examples
Debugging a bug
Think step by step before answering. Our VehicleInfo app marks challans as paid via Razorpay webhook. We see ~3% of payments marked paid in Razorpay but challan status stays PENDING in our DB next day. Walk through every possible reason this gap happens and how to diagnose each one.
Architecture decision
Think step by step before answering. We need to choose between keeping a monolith vs splitting into microservices for VehicleInfo. We have a 3-person backend team, 200k users, 500 payments/day. Walk through the trade-offs, risks at each stage, and give me a recommendation with your reasoning.
Product strategy
Think step by step before answering. We want to add a fleet management feature to VehicleInfo for companies with 10+ vehicles. Think through: who our target users are, what their biggest pains are, what we should build first, what success looks like in 6 months, and what could go wrong.
03

Specify output format explicitly

Claude mirrors your format unless told otherwise. Saying exactly what you want — table, JSON, numbered steps, prose — eliminates reformatting work entirely.

3 ready-to-use examples
Ask for a table
Compare these 3 approaches for storing challan payment status: (A) status column on challans table, (B) separate payments table, (C) event sourcing. Return as a markdown table with columns: Approach | Query Speed | Write Complexity | Consistency | Best For.
Ask for JSON
Design the API response schema for GET /challans/:id that includes challan details, payment status, and vendor assignment info. Return as a JSON object with realistic example values. Include all fields we'd need in the mobile app UI.
Ask for numbered steps
Explain how Razorpay webhooks work end-to-end for a payment flow. Return as numbered steps only — no headers, no paragraphs. Each step should be 1-2 sentences. Include the failure path steps too.
04

XML tags for complex prompts

Use <context>, <task>, <constraints>, <format> tags to separate sections. Claude parses structured prompts with significantly higher accuracy than plain paragraphs.

3 ready-to-use examples
API flow design
<context>VehicleInfo app — users view and pay challans via Razorpay. Node.js, PostgreSQL, AWS. ~500 payments/day.</context> <task>Design the complete backend flow for Challan Payment including API endpoints, DB state machine, webhook handling, and vendor assignment.</task> <constraints> - Payment initiation must be idempotent (no double charges) - Vendor assignment only after webhook confirms success - All state changes logged in payment_events table </constraints> <format>Numbered steps. Failure paths after each step. JSON tables for API contracts.</format>
Code review
<context>Node.js + PostgreSQL fintech app. This is our webhook handler for Razorpay payment events.</context> <task>Review this code for bugs, missing error handling, and security issues.</task> <constraints> - Flag only real issues, not style preferences - Rank each issue: Critical / High / Medium </constraints> <code> [PASTE YOUR CODE HERE] </code> <format>Bullet list: Issue | Severity | Fix</format>
PRD writing
<context>VehicleInfo app, 200k users, B2C vehicle management. We currently show challans but users can't pay in-app.</context> <task>Write a PRD for the Challan Payment feature.</task> <constraints> - No fluff — engineering-PM tone - Max 2 sentences per section point - Success metrics must have specific numbers </constraints> <format>Sections: Problem Statement, Success Metrics, User Stories (4), Scope, Edge Cases, Open Questions</format>
05

Audience + depth specification

The same question to different audiences produces completely different answers. Always specify who will read this and at what technical depth.

3 ready-to-use examples
Non-technical CEO
Explain why we need idempotency keys in our payment system to a non-technical CEO who knows nothing about APIs. Use a simple real-world analogy. Max 150 words. No technical jargon.
Junior engineer
Explain idempotency keys for payment APIs to a junior backend engineer joining our team. Assume they know Node.js and PostgreSQL but have never worked on payments. Include a simple code example and explain what happens if we don't implement it.
Senior engineer
Explain the trade-offs between Redis-based and PostgreSQL-based idempotency key storage for a payment system handling 500 transactions/day. Assume senior engineer audience. Skip the basics — go straight to edge cases: concurrent requests, key expiry, distributed systems failures.
06

Negative constraints

Telling Claude what NOT to do is as powerful as telling it what to do. It eliminates entire categories of unwanted output.

3 ready-to-use examples
No fluff
Explain how to implement webhook signature verification for Razorpay in Node.js. Do NOT include: an introduction paragraph, explanations of what webhooks are, third-party libraries other than crypto, or production recommendations. Just the implementation code with inline comments.
No hedging
Should VehicleInfo store challan payment state in the challans table or a separate payments table? Do NOT say "it depends" or give me options without recommending one. Give me your specific recommendation for our context: Node.js, PostgreSQL, 500 payments/day, 3-person team.
No generic advice
What are the biggest risks in our challan payment flow? Context: Razorpay, Node.js, PostgreSQL, webhook-based status update, vendor assignment post-payment. Do NOT give generic payment advice. Only flag risks specific to our architecture. Do NOT suggest moving to a different payment provider.

Power Techniques

07

Few-shot prompting (examples first)

Give Claude 1–2 examples of your exact ideal output before asking the real question. Claude instantly learns your format, tone, and depth — the most reliable output-control technique.

3 ready-to-use examples
Format a status report
Write a weekly engineering status in this exact format: EXAMPLE OUTPUT: ✅ Done: Deployed payment webhook handler to staging ⚠️ Blocked: Razorpay sandbox credentials expired — waiting on ops 📅 Next: Integrate vendor assignment API, write integration tests Now write the status for: We finished the challan listing API, discovered a race condition in concurrent payments that needs fixing, and next week we're implementing the payment initiation endpoint.
Ticket writing style
Write a Jira ticket in this format: EXAMPLE: Title: [BACKEND] Add idempotency to payment initiation Context: Users retrying failed payments cause duplicate charges Acceptance Criteria: - Given same request with same idempotency key, return same response - Given duplicate key with different body, return 422 - Add integration test covering both cases Now write a ticket for: The challan status is not updating after Razorpay webhook arrives in peak hours — suspected DB deadlock.
Postmortem format
Write a postmortem in this format: EXAMPLE: Summary: 3% of payments showed paid in Razorpay but PENDING in DB for 18 hours. Root Cause: DB deadlock on challans table under concurrent webhook processing. Impact: 47 affected users, 0 financial loss (payments were captured correctly). Fix: Added SELECT FOR UPDATE SKIP LOCKED to webhook handler. Prevention: Added deadlock detection alert + load test before next deploy. Now write a postmortem for: Our challan listing API went down for 45 minutes yesterday because a bad DB migration added a full-table-scan query that ran every 5 seconds.
08

Challenge your own assumptions

After presenting any design, explicitly ask Claude to find weaknesses. This surfaces blind spots faster and more honestly than internal team review.

3 ready-to-use examples
Design review
Here is our challan payment flow design: 1. User taps Pay Now → POST /challans/:id/initiate → creates Razorpay order 2. User completes payment in Razorpay SDK 3. Razorpay sends webhook → we update challan status to PAID 4. Cron job runs every 5 min → assigns available vendor to paid challans Play the role of a skeptical senior engineer. What are the 5 biggest weaknesses in this design? What breaks first when we go from 500 to 5000 payments/day?
PRD review
Here is our PRD for the Fleet Management feature: [PASTE YOUR PRD HERE] What are the 3 biggest assumptions we're making that might be wrong? What user stories are we missing? What would a PM from Uber Freight say is wrong with our success metrics?
DB schema review
Here is our payments table schema: CREATE TABLE payments ( id UUID PRIMARY KEY, challan_id UUID REFERENCES challans(id), razorpay_order_id VARCHAR(50) UNIQUE, amount INTEGER, status VARCHAR(20), created_at TIMESTAMPTZ ); What is missing? What will break when we need to handle refunds, partial payments, or audits? What indexes are we going to regret not having at 10M rows?
09

Multiple options + trade-offs → force a recommendation

Ask for 3 options with trade-offs, then push for a specific recommendation for your context. Never accept "it depends" without a follow-up.

3 ready-to-use examples
Storage decision
Give me 3 approaches for handling challan payment state in VehicleInfo with trade-offs for query speed, consistency, and implementation complexity. Context: Node.js + PostgreSQL, ~500 payments/day, status queryable by vehicle_number in under 100ms, 3-engineer team. Then give me your specific recommendation for our scale and constraints. No hedging.
Notification architecture
Give me 3 approaches for real-time payment status notifications in our mobile app with trade-offs for latency, battery drain, and server complexity. Options to consider: WebSockets, polling, push notifications via Firebase. Context: React Native app, Node.js backend, 5k daily active users. Then recommend one for our scale.
Infrastructure decision
Give me 3 deployment options for VehicleInfo backend with trade-offs for cost, operational complexity, and scaling ease. Options: Single EC2 + PM2, Docker on ECS Fargate, Kubernetes on EKS. Context: 3-person team, $2k/month infra budget, need to scale from 200k to 2M users over 2 years. Give me a specific recommendation.
10

Flag uncertainty explicitly

Add Rate your confidence High/Medium/Low per section. Flag anything uncertain. Claude will mark the gaps — you know exactly what to verify before acting on the output.

3 ready-to-use examples
Technical fact-check
Explain how Razorpay handles webhook retry logic — how many retries, at what intervals, and what happens after all retries are exhausted. Rate your confidence (High/Medium/Low) for each fact. Flag anything you're not certain about so I know what to verify in the Razorpay docs.
Compliance check
What RBI regulations apply to VehicleInfo if we add in-app challan payments through a payment aggregator? Rate your confidence per regulation. Flag anything that may have changed after your training cutoff — I'll verify those with a compliance consultant.
Architecture estimate
How many concurrent users can a single Node.js process handle on an AWS t3.medium instance for our payment API — which does a DB query + Razorpay API call per request? Give me a ballpark estimate. Rate your confidence (High/Medium/Low) and tell me exactly what assumptions you're making so I can validate with a load test.
11

Simulate stress tests on your system

Ask Claude to walk through failure modes as load increases. This surfaces architectural weaknesses before they hit production.

3 ready-to-use examples
Load scaling
Walk through what breaks in our VehicleInfo payment system as we scale from current 500 payments/day → 5,000 → 50,000. At each level: what fails first, why, and what we'd need to change. Our stack: Node.js, PostgreSQL RDS t3.medium, no caching layer, synchronous Razorpay webhook processing.
Failure scenarios
Walk me through every failure scenario in this payment flow: User pays → Razorpay captures → webhook fires → we update DB → assign vendor At each step: what can go wrong, what the user experiences, whether money is lost or double-charged, and what our recovery path is. Be specific about which failures are recoverable vs catastrophic.
Infrastructure failure
Simulate what happens to VehicleInfo if our primary PostgreSQL RDS instance goes down during peak payment hours (10–11 AM, ~50 concurrent payments in-flight). Walk through: what users experience, what data might be corrupted, how Razorpay webhooks are affected, what our recovery steps should be, and what we should build now to make this recoverable.
12

Extended Thinking mode

Enable in claude.ai settings or add Take time to reason through this thoroughly before responding. Produces qualitatively different, deeper answers — not just more words, but new insights.

3 ready-to-use examples
Hard architecture choice
Take time to reason through this thoroughly before responding. VehicleInfo needs to decide: should we implement Event Sourcing for our payment ledger, or stay with traditional CRUD + audit log? We process 500 payments/day, need full audit trail for compliance, have a 3-person team, and are on PostgreSQL. Think through all trade-offs, our constraints, and long-term implications. Give me a final recommendation I can present to our CTO.
Tricky bug
Take time to reason through this thoroughly before responding. We have a race condition in our challan payment flow. Two webhook events for the same payment_id arrive within 100ms of each other. Our Node.js handler reads the current status, checks if it's already PAID, and if not — updates it. We're seeing duplicate vendor assignments. Think through all the ways this can fail and give me the correct fix using PostgreSQL row-level locking.
Product strategy
Take time to reason through this thoroughly before responding. VehicleInfo has 200k registered users, 50k monthly active. We're deciding between: (A) add fleet management for commercial vehicle operators, (B) expand to insurance integration, (C) deepen the payment/compliance features. We have 6 months runway, a team of 8, and need to show growth to raise Series A. Think through each option deeply and give me your recommended bet with full reasoning.
13

Projects for persistent context

Set up a Claude Project once with your role, stack, and company context. Every future session inherits it — you never repeat context again. Highest ROI setup you can do.

3 ready-to-use system prompts — paste into Claude Projects
Backend engineer
You are a senior backend engineer and technical advisor for VehicleInfo — a vehicle management app where users view and pay challans. Stack: Node.js, PostgreSQL 14, AWS (EC2 + RDS), Razorpay, Redis for caching. Team: 3 backend engineers, 1 PM, 1 DevOps. Always: think step by step, flag edge cases, give specific recommendations (not "it depends"), and tie everything to our VehicleInfo context. Never give generic advice.
Product manager
You are a senior PM and product thinking partner for VehicleInfo — B2C vehicle management app, 200k users, India market. Key features: challan viewing, challan payment via Razorpay, vendor assignment. Business model: transaction fee on payments + B2B fleet management SaaS. Always: structure your thinking, give opinionated recommendations, use Indian market context, and assume I understand product basics — skip the fundamentals.
Full-stack context
You are a technical co-founder and advisor for VehicleInfo. Product: Vehicle management app — challan viewing, payment, vendor assignment, fleet management (coming soon). Stack: React Native (mobile), Node.js + Express (API), PostgreSQL 14 (DB), Razorpay (payments), AWS. Team: 8 people, 6 months runway, targeting Series A. Give me advice like a CTO who cares about shipping fast AND building right. Always call out shortcuts that will hurt us later.
14

Iterative depth drilling

First response is always a draft. Push with specific context to go deeper — each turn adds a layer of precision you can't get in one shot.

3 follow-up pushes — use after any first response
Too generic
This is too generic. Make it specific to VehicleInfo: Node.js, PostgreSQL, Razorpay, 500 payments/day, 3-engineer team. Remove any advice that doesn't apply to our exact constraints.
Missing edge cases
Good — now add the failure scenarios. What happens when: (1) the Razorpay webhook arrives twice, (2) the DB is under deadlock pressure, (3) the vendor assignment service is down but payment succeeded. Show the recovery path for each.
Need the code
Good explanation. Now give me the actual implementation — Express middleware, PostgreSQL queries, and error handling. Use our stack: Node.js 18, pg library (not Prisma), async/await. Include the idempotency check in the same transaction as the status update.

What Kills Output Quality

Anti-Patterns to Avoid

A small set of repeatable mistakes caps most users at 60% quality. Eliminating these doubles output quality with zero extra effort.

⚠️

Read this first

Before rating any Claude response as poor — audit your prompt against these anti-patterns first. Most "Claude failures" are actually prompt failures.

✗

Vague open-ended prompts

"Tell me about payments" → Wikipedia overview. Always specify: which aspect, what depth, what format, for what purpose. Vague in = generic out. Every single time.

✗

Trusting without verification

Claude is confident even when wrong. Never ship code, data, or decisions based solely on Claude's output. Validate technical facts, APIs, regulations, and numbers independently.

✗

No context between sessions

Claude has zero memory between chats. Use Projects for persistent context, or always re-establish role + constraints at the start of every session. This is non-negotiable for quality.

✗

Accepting the first response as final

Real value comes from iteration. Push back: "Make this specific to X", "What edge cases are missing?", "Give me the opposite view." First responses are always drafts, never finals.

✗

Overloading one prompt

5 questions in one message → Claude addresses all superficially. One clear goal per interaction. Use follow-ups for depth. Separate concerns into separate messages.

✗

Not asking for edge cases

Claude defaults to the happy path. For any design or flow, always add: What could go wrong? What failure modes am I missing? What are the edge cases I haven't considered?

✗

Accepting "it depends" without drilling

When Claude hedges, force a concrete answer: Given my context is X and constraint is Y, which specific approach do you recommend and why? Never let it off the hook with non-answers.

✗

Using Claude as a search engine for current info

Claude has a knowledge cutoff. For current regulations, recent API changes, or fast-moving topics — use built-in web search or verify with official docs. Don't trust training data for time-sensitive facts.
